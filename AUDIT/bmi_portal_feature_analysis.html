BMI portal vs world-class UMS — side by side

BMI portal — what it does well
  • ✅100% open-source, self-hosted, MIT-licensed stack — strong sovereignty stance
  • ✅Local LLM (Ollama + Llama 3.2) — no data leaves the server
  • ✅JWT auth + RBAC (Admin, Registrar, Staff) via PocketBase
  • ✅AES-256 encryption key planned; bcrypt password hashing
  • ✅QR code certificate generation with content hashing
  • ✅Audit logging for all actions
  • ✅Typed frontend (TypeScript 92%) with React 19 + Vite
  • ✅Automatic HTTPS via Caddy
  • ✅Docker Compose + Makefile for reproducible dev setup
  • ✅Extensive planning docs (README, ARCHITECTURE, SECURITY, CHANGELOG)
  • ✅Rate limiting middleware via Hono.js + Helmet headers
  • ✅Litestream for SQLite backup to S3-compatible storage
  • ✅CONTRIBUTING.md + SECURITY.md with responsible disclosure process
World-class standard — where BMI falls short
  • ❌No automated test suite (unit, integration, e2e) — 0 test files visible
  • ❌No GitHub Actions CI pipeline for lint/test/build on every PR
  • ❌No WCAG 2.1 accessibility compliance or audits
  • ❌No internationalisation (i18n) — English-only UI
  • ❌No OpenAPI / Swagger spec for the Hono.js API
  • ❌SQLite write-concurrency ceiling will hurt under load
  • ❌No MFA / 2FA for users (critical for institutional systems)
  • ❌Missing: timetabling, grading/rubrics, learning outcomes, LTI/LMS integration
  • ❌No structured student portal (self-service grade lookup, course registration)
  • ❌No observability stack (metrics, distributed traces, alerting)
  • ❌Dependency vulnerability scanning (Dependabot) referenced but not configured
  • ❌No GDPR/data-subject request workflow despite privacy-first claim
  • ❌Many root-level fix/restart scripts suggest brittle deployment
